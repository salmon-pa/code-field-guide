Part 1 · Orientation

The map: how many languages, and which does what

There are 700+ programming languages. Only about ten run the professional world — and only two matter for you. Here's the whole landscape on one screen, so you know what you're skipping and why.

Python · scripting · automation · AI

Your core

Automations, AI & LLM orchestration, data pulls, scrapers, backends. Salmon and most of your automation scripts are Python. This is the language you learn to read.

JavaScript / TypeScript · web & dashboards

Recognize

Everything that runs in a browser: websites, buttons, Mission Control and your dashboards. TypeScript is JavaScript with safety rails. You learn to recognize it, not master it.

SQL · databases

Recognize

Not a "programming" language — a question language for databases. "Give me all clients where spend > X." Worth recognizing; it reads almost like English.

HTML & CSS · structure & style

Recognize

HTML is the skeleton of a web page, CSS is the paint. Not logic — layout. You'll see them constantly; you don't need to write them.

JSON & YAML · data & config

Recognize

Not code — the format data travels in. Every API answer, every settings file. Once you can read JSON, half of "technical" stops being scary.

Swift · Kotlin · mobile apps

Skip

iPhone (Swift) and Android (Kotlin) apps. Not part of your AI-automation world. Ignore unless you build a mobile app.

C · C++ · Rust · Go · systems & speed

Skip

Operating systems, games, high-performance engines. Powerful, slow to learn, zero relevance to directing an AI service. Know the names, move on.

Java · PHP · Ruby · C# · older business backends

Skip

Run huge parts of the corporate web. Fine languages — just not yours. If a vendor mentions them, now you know where they sit.

The one thing to remember: languages split by job, not by "good vs bad." Python = automation & AI (yours). JavaScript = anything in a browser (your dashboards). Everything else is someone else's job to worry about.

Part 2 · The pick

Why Python is your one language

You direct an AI-automation team. Every automation follows the same shape — pull data, hand it to an AI, do something with the answer — and Python is the language that shape is written in. Learn to read Python and you can follow 90% of what your team ships. A little JavaScript literacy on top lets you follow the dashboard side. That's the whole curriculum.

Part 3 · The sprint

Seven days, look-and-do

Each day: one idea, one thing you actually do, one code sample to read, and a thought to chew on while driving. Check a day off when the "do this" is done. Your progress saves automatically on this device.

Day 1

The shape of code

~15 min

Grasp this: every program, no matter how scary it looks, does three things in order — get something in → do something to it → send something out. That's it. Find those three parts and the fear drops.

# get something in
clients = read_sheet("active_clients")

# do something to it
at_risk = [c for c in clients if c.spend < c.target]

# send something out
send_alert(at_risk)
Do this now Open any one script your team or an AI agent wrote. Don't try to understand every line — just find the in, the do, and the out. Say each one out loud.

🚗While driving: think of one automation you use daily. What goes in, what comes out? You just described its code.

Day 2

The nouns — variables & data

~15 min

Grasp this: a variable is just a labelled box holding a value. The main kinds: text ("Acme Co"), numbers (500000), lists [a, b, c], and dictionaries — labelled pairs, like a mini record.

client   = "Acme Co"      # text (a string)
budget   = 500000         # number
channels = ["Meta", "Google"]  # list
report   = { "leads": 120, "sales": 45 }  # dictionary
Do this now In a real script, point at 5 variables. For each, say: text, number, list, or dictionary? That's the whole skill.

🚗While driving: a client record — name, budget, list of channels — is text + number + list living together. That's a "dictionary." You already think in them.

Day 3

The verbs — logic & loops

~20 min

Grasp this: code makes decisions with if / else ("if spend is low, alert") and repeats work with a loop ("for each client, check…"). A function is a named recipe you can reuse. Three ideas, and you can read logic.

def check_client(client):      # a reusable recipe
    if client.spend < client.target:
        return "at risk"
    else:
        return "healthy"

for c in clients:          # repeat for every client
    print(c.name, check_client(c))
Do this now Find one if/else and one loop in a real script. Read the condition out loud in plain English: "if _____, then _____." If you can do that, you can review logic.

🚗While driving: "for every red light, stop" is a loop with an if. Code borrowed your everyday logic — it just spells it out.

Day 4

The pipes — APIs & secrets

~20 min

Grasp this: an API is how one program asks another for data — "Meta, give me yesterday's spend." The answer comes back as JSON. Access needs an API key (a password) — and keys must never sit in the code. Remember that last part; it's your #1 review flag.

key = os.environ["META_API_KEY"]   # ✓ key from a safe vault
data = requests.get("https://api.meta.com/spend",
                    headers={"auth": key})
spend = data.json()["total"]        # read the JSON answer
Do this now Find one API call in your codebase (search for requests, fetch, or http). Spot the URL (who it's asking) and the key (how it's allowed). Then check: is that key hidden, or sitting in plain text?

🚗While driving: an API is a drive-through. You ask through the speaker (URL), show your card (key), get your order (JSON). Simple as that.

Day 5

The AI-automation shape

~15 min

Grasp this: almost every AI automation you own is the same five steps — get data → build a prompt → ask the AI → read the answer → store or send it. Once you see this shape, you understand your own service from the inside.

data   = read_sheet("client_spend")         # 1 get data
prompt = f"Summarize this for the client: {data}" # 2 build prompt
answer = ai.ask(prompt)                    # 3 ask the AI
text   = answer.content                     # 4 read the answer
send_email(text)                            # 5 send it
Do this now Take one Salmon automation and map it onto these five steps out loud. Where does its data come from? What does it ask the AI? Where does the answer go? That's an architecture review.

🚗While driving: your morning brief = read calendar/email → ask AI to summarize → send to you. Five steps. You've been running this shape all along.

Day 6

The reviewer's eye

~20 min

Grasp this: you don't judge if a line is "correct" — you judge whether the code is safe, clear, and maintainable. Five things a good reviewer looks for, no coding required. Use the cheat sheet below on real code.

Do this now Take one real script and run the Reviewer's cheat sheet (next section) against it. Write down 3 things you'd flag. You're now reviewing — not reading.

🚗While driving: reviewing code is like reviewing a media plan — you're not redoing the math, you're asking "is this safe, clear, and will it hold up?"

Day 7

Give real feedback

~25 min

Grasp this: the payoff. You read a real piece of code, form your own opinion, then pressure-test it against a second opinion (me). When your instinct matches, you've arrived.

Do this now Pick a real script or pull request. Write 3 review comments in your own words. Then paste it to Claude with "grade this code and tell me what you'd flag" and compare. Where you agreed, that's your new skill. Where you missed, that's tomorrow's lesson.

🚗While driving: you already do this with decks and pitches — read, judge, give sharp notes. Code is just a new medium for the same instinct.

Part 4 · Keep this open

The reviewer's cheat sheet

Six questions that make your feedback land — and make your dev team take you seriously. No syntax knowledge needed. Ordered by how much they matter.

Critical"Any secrets in the code?" API keys, passwords, tokens should live in a vault (env variables) — never typed into the file. This is the #1 thing that gets companies hacked.
Critical"What happens when this fails?" If the API is down or data is missing — does it crash loudly, or fail silently and send bad numbers? Silent failure is worse.
High"Is this hardcoded?" A budget or date typed straight into the code means editing code to change it. Ask if it should be a setting instead.
High"Is this duplicated anywhere?" The same block copy-pasted in five places means five places to fix later. "Can this be one reusable function?"
Worth asking"Will it hold at 10× the data?" Fine for 10 clients — what about 500? Cheap to ask now, expensive to discover in production.
Worth asking"Will someone understand this in 6 months?" Clear names, a comment on the tricky part. If you can't follow it, that's legitimate feedback — not your ignorance.

Part 5 · Your cheat code

Turn Claude into your private tutor

You have a coding teacher on call 24/7 — me. You'll learn 3× faster from your own real code than any course. Copy any of these, paste it with a script, and go.

Plain-English walkthrough
Explain this code to me in plain English, line by line, like I'm a smart CEO who doesn't code. Tell me what goes in, what it does, and what comes out.
Grade & flag

Check my instinct

What could go wrong
Walk me through everything that could go wrong when this code runs — bad input, API down, missing data. For each, does the code handle it or break?
